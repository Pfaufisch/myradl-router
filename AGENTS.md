# Repository Guidelines

## Project Structure & Module Organization

This is a browser-only React 19, TypeScript, and Vite SPA.

- `src/App.tsx` contains the screen flow and top-level state.
- `src/hooks/` contains browser-sensor hooks, currently compass and geolocation guidance.
- `src/lib/` contains focused API, persistence, and geometry modules. Keep network normalization and calculations out of UI components.
- Tests are colocated with their implementation as `*.test.ts` or `*.test.tsx`.
- `src/styles.css` provides the global mobile-first design system; `src/test/setup.ts` configures DOM matchers.
- `dist/` is generated by Vite and must not be committed.

## Build, Test, and Development Commands

- `npm install` installs the locked dependency set.
- `npm run dev` starts the Vite development server with hot reload.
- `npm test` runs all Vitest suites once.
- `npm run test:watch` reruns relevant tests during development.
- `npm run build` type-checks with TypeScript and creates the production bundle in `dist/`.
- `npm run preview` serves the production bundle locally.

Run `npm test && npm run build` before submitting changes.

## Coding Style & Naming Conventions

Use strict TypeScript, ES modules, two-space indentation, single quotes, and trailing commas in multiline structures. Components and exported types use `PascalCase`; hooks use `useCamelCase`; functions and variables use `camelCase`; constants use `UPPER_SNAKE_CASE`. Prefer small pure functions for geometry and API transformation. No formatter or linter is configured, so follow the surrounding style and keep imports tidy.

## Testing Guidelines

Vitest, jsdom, React Testing Library, and `jest-dom` are used. Add regression tests for every bug, especially API response variations, persistence, distance/bearing math, and permission fallbacks. Mock external APIs and browser sensors; tests must not depend on live network access. There is no enforced coverage threshold, but new branches should be exercised.

## Commit & Pull Request Guidelines

History is small; existing subjects are short and action-oriented. Prefer imperative messages such as `Fix house-number address lookup`. Pull requests should explain user-visible behavior, list verification commands, link relevant issues, and include mobile screenshots for UI changes. Call out changes to external API assumptions or permission handling.

## Security & Configuration

Do not add private API keys or user-location logging. The app intentionally has no backend. Geolocation and device orientation require HTTPS outside localhost; preserve clear denied-permission fallbacks and OpenStreetMap attribution.
